### 2.2 Overall 公式

```
Overall = (文本准确率 + CDM准确率 + TEDS准确率) / 3

其中:
  文本准确率 = (1 − Text Edit-dist) × 100
  CDM准确率  = Formula CDM × 100
  TEDS准确率 = Table TEDS × 100
```

- **验证：** (96.7 + 97.49 + 94.76) / 3 = 96.32 ≈ 96.33 ✅
- **我们的：** (96.5 + 94.4 + 93.0) / 3 = 94.63

## 三、精度差距分析（待排查）

### 3.1 差距分解

| 因子 | 贡献 | 说明 |
|---|---|---|
| Text Edit | -0.2 pt | 0.035 vs 0.033，文本提取本身接近官方 |
| Formula CDM | **-3.05 pt** | 0.944 vs 0.975，最大差距来源（占 Overall 差距 60%） |

| Reading Order | -0.3 pt | 0.130 vs 0.127，差距最小 |

**结论：Overall 偏低的主要原因是 CDM 和 TEDS 差距叠加。**

### 3.2 已知假设（按可能性排序）

**假设 A：llama.cpp 版本差异（最可能）**
- 我们用了 b9884（用户手动下载），原计划是 b9637
- 不同 llama.cpp 版本可能有不同的推理行为
- 测试：用 b9637 重新跑 adapter，对比评分

**假设 B：GGUF 量化差异**
- 我们用的是从 ModelScope 下载的 GGUF（Q4_K_M 量化？）
- 官方 PaddleOCR-VL-1.6 用的是 BF16 未量化模型
- 量化会导致精度损失，尤其对公式和表格影响大
- 测试：检查 GGUF 量化级别，尝试更高精度的量化

**假设 C：VLM 推理参数差异**
- 当前参数：`--temp 0 --top-k 1 --seed 1 --flash-attn on`
- 官方可能使用不同的采样策略
- 测试：调整 top-k、temperature 等参数

**假设 D：layout 模型在 CPU 上运行**
- 当前运行的 adapter 启动于 DirectML 修复**之前**
- layout 检测在 CPU 上，可能影响版面分割精度
- 影响：layout 不准 → table/formula 定位偏差 → TEDS/CDM 降低
- 测试：修复后重新跑 adapter

**假设 E：`\mathcolor` 渲染覆写（CDM 专项）**
- CDM 代码中 `\DeclareDocumentCommand{\mathcolor}` 覆写了 xcolor 的 mathcolor
- 这个覆写可能改变公式渲染颜色，影响 CDM 颜色匹配算法
- 已确认 CDM F1=1.0（相同公式测试通过），但不同公式可能有细微差异
- 测试：对比官方 Paddle 原生路径的 CDM 输出

### 3.3 排查优先级

1. **先用 DirectML 修复后的 adapter 重新跑**（假设 D）— 零成本，只是重跑
2. **换 b9637 llama.cpp 版本**（假设 A）— 需要下载对应版本
3. **检查 GGUF 量化级别**（假设 B）— 一行命令
4. **调整 VLM 参数**（假设 C）— 可能需要参数扫描

### 4.2 服务状态

```powershell

  -ArgumentList '--host','127.0.0.1','--port','8111',`
    '-m','<full-path-to>\PaddleOCR-VL-1.6-GGUF.gguf',`
    '--mmproj','<full-path-to>\PaddleOCR-VL-1.6-GGUF-mmproj.gguf',`
```

## 五、WSL 注意事项

- **路径转换：** 在 Git Bash 中调用 WSL 需要 `MSYS_NO_PATHCONV=1` 前缀
- **PATH 污染：** WSL 内用 `export PATH=/usr/local/texlive/2026/bin/x86_64-linux:/usr/local/bin:/usr/bin:/bin` 避免 Windows PATH 泄漏
- **CRLF：** `.sh` 文件必须 LF 行尾（已在 `.gitattributes` 中配置）
